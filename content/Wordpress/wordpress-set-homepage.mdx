## Overview

This guide covers WordPress URL configuration—a critical aspect of WordPress administration that affects site accessibility, SEO, asset loading, and deployment workflows. Understanding these concepts is essential for system administrators, DevOps engineers, and developers working with WordPress in production environments.

### URL Resolution Order

WordPress resolves site URLs using the following precedence (highest to lowest):

1. **Constants in wp-config.php** (`WP_HOME`, `WP_SITEURL`)
2. **Filtered values** via `home_url` and `site_url` filters
3. **Database options** (`home`, `siteurl` in `wp_options` table)
4. **Installation defaults** (set during WordPress setup)

Understanding this hierarchy is crucial for debugging URL-related issues and implementing proper multi-environment configurations.

---

### Define URL

WordPress uses two core constants to manage site URLs: `WP_HOME` and `WP_SITEURL`. These constants override the values stored in the database and are essential for deployment configurations, migrations, and containerized environments.

**Technical Details:**

- **WP_HOME**: The URL visitors use to access your site (front-end)
- **WP_SITEURL**: The URL where WordPress core files are located
- Both constants prevent URL changes via wp-admin (Settings → General)
- Defined in `wp-config.php` or as environment variables
- Take precedence over `home` and `siteurl` options in `wp_options` table

**Internal WordPress Functions:**

| Function | Uses | Returns |
|----------|------|---------|
| `home_url()` | `WP_HOME` | Front-end URL for visitors |
| `site_url()` | `WP_SITEURL` | WordPress installation URL |
| `admin_url()` | `WP_SITEURL` | Admin area URL (`/wp-admin/`) |
| `content_url()` | `WP_SITEURL` | Content directory URL (`/wp-content/`) |
| `includes_url()` | `WP_SITEURL` | Includes directory URL (`/wp-includes/`) |
| `plugins_url()` | `WP_SITEURL` | Plugins directory URL |

**URL Anatomy Example:**

```
https://example.com/blog/wp-admin/edit.php
├─────────────────┘    └───────────────────┘
     WP_HOME            Path after WP_SITEURL

WP_HOME:    https://example.com
WP_SITEURL: https://example.com/blog
```

```php copy
WORDPRESS_CONFIG_EXTRA:
               define('WP_HOME','http://localhost');
               define('WP_SITEURL','http://localhost');
```

### Configuration Methods

#### 1. wp-config.php (Static)

Add directly to `wp-config.php` before the "That's all, stop editing!" comment:

```php

define('WP_SITEURL', 'https://example.com');
```

**File Location Reference:**

```plaintext

├── wp-config.php          ← Define constants here
├── wp-settings.php        ← Loaded after wp-config.php
├── wp-content/
├── wp-admin/
└── wp-includes/
```

**Placement in wp-config.php:**

```php
<?php
// Database settings
define('DB_NAME', 'wordpress');
define('DB_USER', 'root');
define('DB_PASSWORD', 'password');

// URL Constants - ADD HERE (before authentication keys)

// ...

/* That's all, stop editing! Happy publishing. */
```


#### 3. Subdirectory Installation

When WordPress core is in a subdirectory:

```php
define('WP_HOME', 'https://example.com');
define('WP_SITEURL', 'https://example.com/wordpress');
```

**Directory Structure for Subdirectory Installation:**

```plaintext
public_html/                        ← Document root (WP_HOME)
├── index.php                       ← Modified loader
├── .htaccess                       ← Rewrite rules
└── wordpress/                      ← WordPress core (WP_SITEURL)
    ├── wp-admin/
    ├── wp-content/
    ├── wp-includes/
    ├── wp-config.php
    └── index.php
```

**Modified Root index.php:**

```php
<?php
/**
 * Front to the WordPress application.
 * Modified to load WordPress from subdirectory.
 */
define('WP_USE_THEMES', true);
require(__DIR__ . '/wordpress/wp-blog-header.php');
```

**Root .htaccess Configuration:**

```apache
# BEGIN WordPress Subdirectory
<IfModule mod_rewrite.c>
RewriteEngine On
RewriteBase /
RewriteRule ^index\.php$ - [L]

# Don't rewrite requests to actual files or directories
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d

# Rewrite everything else to index.php
RewriteRule . /index.php [L]
</IfModule>
# END WordPress Subdirectory
```

**Use Cases for Subdirectory Installation:**

| Scenario | WP_HOME | WP_SITEURL |
|----------|---------|------------|
| Clean URLs (core hidden) | `https://example.com` | `https://example.com/wp` |
| Multiple apps on domain | `https://example.com/blog` | `https://example.com/blog/wordpress` |
| Development isolation | `http://localhost:8080` | `http://localhost:8080/wordpress` |

### Database Direct Update (Migration/Emergency)

When `wp-config.php` is inaccessible, update URLs via SQL:

```sql
-- Update site URL
UPDATE wp_options
SET option_value = 'https://newdomain.com'
WHERE option_name IN ('home', 'siteurl');

-- Verify changes
SELECT option_name, option_value
FROM wp_options
WHERE option_name IN ('home', 'siteurl');
```

**Warning**: This method doesn't update serialized data in post content or postmeta. Use WP-CLI or Search Replace DB for full migrations.

**Database Schema Context:**

The `wp_options` table stores WordPress configuration as key-value pairs:

```sql
-- wp_options table structure
+-------------+---------------------+------+-----+---------+----------------+
| Field       | Type                | Null | Key | Default | Extra          |
+-------------+---------------------+------+-----+---------+----------------+
| option_id   | bigint(20) unsigned | NO   | PRI | NULL    | auto_increment |
| option_name | varchar(191)        | NO   | UNI |         |                |
| option_value| longtext            | NO   |     |         |                |
| autoload    | varchar(20)         | NO   |     | yes     |                |
+-------------+---------------------+------+-----+---------+----------------+
```

**Complete Migration Query Set:**

```sql
-- Start transaction for safety
START TRANSACTION;

-- Update core URL options
UPDATE wp_options
SET option_value = REPLACE(option_value, 'http://olddomain.com', 'https://newdomain.com')
WHERE option_name IN ('home', 'siteurl');

-- Update GUID in posts (permanent URLs)
UPDATE wp_posts
SET guid = REPLACE(guid, 'http://olddomain.com', 'https://newdomain.com');

-- Update post content (images, links)
UPDATE wp_posts
SET post_content = REPLACE(post_content, 'http://olddomain.com', 'https://newdomain.com');

-- Update post excerpts
UPDATE wp_posts
SET post_excerpt = REPLACE(post_excerpt, 'http://olddomain.com', 'https://newdomain.com');

-- Verify changes before committing
SELECT option_name, option_value FROM wp_options WHERE option_name IN ('home', 'siteurl');

-- Commit if everything looks correct
COMMIT;

-- Or rollback if issues found
-- ROLLBACK;
```

**Serialized Data Warning:**

WordPress stores complex data as PHP serialized strings. Direct SQL replacement breaks serialization:

```php
// Example serialized data in wp_options
a:2:{s:4:"home";s:24:"http://olddomain.com";s:7:"version";s:5:"6.4.2";}
//   ↑                    ↑
//   String length (24)   Actual string

// After naive replace (BROKEN):
a:2:{s:4:"home";s:25:"https://newdomain.com";s:7:"version";s:5:"6.4.2";}
//   ↑                    ↑
//   Still 24!            Now 25 chars - LENGTH MISMATCH
```

This is why WP-CLI `search-replace` is strongly recommended for production migrations.

### WP-CLI Method

Recommended for migrations and server-side automation:

```bash
# Update URLs
wp option update home 'https://newdomain.com'
wp option update siteurl 'https://newdomain.com'

# Search-replace across all tables (handles serialized data)
wp search-replace 'http://olddomain.com' 'https://newdomain.com' \
  --all-tables --report-changed-only

# Flush cache and rewrite rules
wp cache flush
wp rewrite flush
```

**WP-CLI Search-Replace Options:**

| Flag | Description |
|------|-------------|
| `--dry-run` | Preview changes without applying |
| `--all-tables` | Include non-WordPress tables |
| `--network` | For multisite, run on all sites |
| `--skip-columns=<cols>` | Skip specific columns (e.g., guid) |
| `--precise` | Use PHP serialization for accuracy |
| `--report-changed-only` | Only report tables with changes |
| `--log=<file>` | Log changes to file |

**Comprehensive Migration Script:**

```bash
#!/bin/bash
# WordPress URL Migration Script
# Usage: ./migrate-urls.sh OLD_URL NEW_URL

OLD_URL="${1:-http://staging.example.com}"
NEW_URL="${2:-https://production.example.com}"
BACKUP_FILE="pre-migration-$(date +%Y%m%d-%H%M%S).sql"

echo "=== WordPress URL Migration ==="
echo "From: $OLD_URL"
echo "To:   $NEW_URL"

# Create backup
echo "[1/5] Creating database backup..."
wp db export "$BACKUP_FILE" --add-drop-table

# Dry run first
echo "[2/5] Analyzing changes (dry run)..."
wp search-replace "$OLD_URL" "$NEW_URL" \
  --all-tables \
  --dry-run \
  --report-changed-only

# Confirm before proceeding
read -p "Proceed with migration? (y/N): " confirm
if [[ "$confirm" != "y" && "$confirm" != "Y" ]]; then
  echo "Migration cancelled."
  exit 0
fi

# Execute replacement
echo "[3/5] Executing URL replacement..."
wp search-replace "$OLD_URL" "$NEW_URL" \
  --all-tables \
  --precise \
  --report-changed-only

# Handle protocol changes (http → https)
if [[ "$NEW_URL" == https://* && "$OLD_URL" == http://* ]]; then
  echo "[3.5/5] Fixing mixed protocol references..."
  OLD_HTTP="${OLD_URL/https:/http:}"
  wp search-replace "$OLD_HTTP" "$NEW_URL" \
    --all-tables \
    --precise \
    --report-changed-only
fi

# Flush caches
echo "[4/5] Flushing caches..."
wp cache flush
wp transient delete --all
wp rewrite flush

# Verify
echo "[5/5] Verifying configuration..."
echo "Home URL:    $(wp option get home)"
echo "Site URL:    $(wp option get siteurl)"

echo ""
echo "Migration complete! Backup saved: $BACKUP_FILE"
```

**Multisite Migration:**

```bash
# List all sites in network
wp site list --fields=blog_id,url

# Migrate specific site
wp search-replace 'http://old.com' 'https://new.com' \
  --url=http://old.com \
  --all-tables-with-prefix

# Migrate entire network

  --network \
  --all-tables
```

#### HTTPS/SSL Migration

When moving from HTTP to HTTPS:

```php
// Force HTTPS
define('FORCE_SSL_ADMIN', true);

// Update URLs to HTTPS
define('WP_HOME', 'https://example.com');
define('WP_SITEURL', 'https://example.com');
```

Then update content URLs:

```bash
wp search-replace 'http://example.com' 'https://example.com' --all-tables
```

**SSL-Related Constants Reference:**

| Constant | Purpose | Default |
|----------|---------|---------|
| `FORCE_SSL_ADMIN` | Force HTTPS for wp-admin and wp-login | `false` |
| `FORCE_SSL_LOGIN` | Force HTTPS for login only (deprecated) | `false` |

**Complete HTTPS Migration Checklist:**

```php
// wp-config.php HTTPS configuration
define('FORCE_SSL_ADMIN', true);
define('WP_HOME', 'https://example.com');
define('WP_SITEURL', 'https://example.com');

// Force HTTPS for all content
if (strpos($_SERVER['HTTP_X_FORWARDED_PROTO'] ?? '', 'https') !== false) {
    $_SERVER['HTTPS'] = 'on';
}
```

**.htaccess HTTPS Redirect:**

```apache
RewriteEngine On

RewriteCond %{HTTP:X-Forwarded-Proto} !https

</IfModule>
```


#### Reverse Proxy / Load Balancer

For sites behind proxies (Cloudflare, nginx):

```php
// Trust proxy headers
if (isset($_SERVER['HTTP_X_FORWARDED_PROTO']) && $_SERVER['HTTP_X_FORWARDED_PROTO'] === 'https') {
    $_SERVER['HTTPS'] = 'on';
}

define('WP_HOME', 'https://example.com');
define('WP_SITEURL', 'https://example.com');
```

**Complete Reverse Proxy Configuration:**

WordPress needs to correctly detect the original client request when behind a proxy. Add this to `wp-config.php`:

```php
/**
 * Reverse Proxy / Load Balancer Configuration
 * Place before wp-settings.php require
 */

// Trust X-Forwarded-* headers from known proxies
$trusted_proxies = [
    '10.0.0.0/8',      // Internal network
    '172.16.0.0/12',   // Docker default
    '192.168.0.0/16',  // Local network
    '127.0.0.1',       // Localhost
];

// Check if request is from trusted proxy
$client_ip = $_SERVER['REMOTE_ADDR'] ?? '';
$is_trusted = false;

foreach ($trusted_proxies as $proxy) {
    if (strpos($proxy, '/') !== false) {
        // CIDR notation check (simplified)
        list($subnet, $mask) = explode('/', $proxy);
        if ((ip2long($client_ip) & ~((1 << (32 - $mask)) - 1)) == ip2long($subnet)) {
            $is_trusted = true;
            break;
        }
    } elseif ($client_ip === $proxy) {
        $is_trusted = true;
        break;
    }
}

if ($is_trusted) {
    // Set HTTPS from forwarded protocol
    if (isset($_SERVER['HTTP_X_FORWARDED_PROTO']) && $_SERVER['HTTP_X_FORWARDED_PROTO'] === 'https') {
        $_SERVER['HTTPS'] = 'on';
    }

    // Set real client IP
    if (isset($_SERVER['HTTP_X_FORWARDED_FOR'])) {
        $forwarded_ips = explode(',', $_SERVER['HTTP_X_FORWARDED_FOR']);
        $_SERVER['REMOTE_ADDR'] = trim($forwarded_ips[0]);
    } elseif (isset($_SERVER['HTTP_X_REAL_IP'])) {
        $_SERVER['REMOTE_ADDR'] = $_SERVER['HTTP_X_REAL_IP'];
    }

    // Set original host
    if (isset($_SERVER['HTTP_X_FORWARDED_HOST'])) {
        $_SERVER['HTTP_HOST'] = $_SERVER['HTTP_X_FORWARDED_HOST'];
    }
}

define('WP_HOME', 'https://example.com');
define('WP_SITEURL', 'https://example.com');
```

### Troubleshooting

#### Redirect Loop

Mismatch between WP_HOME and actual URL. Check:
- `.htaccess` rewrite rules
- Proxy/CDN configuration
- `$_SERVER['HTTPS']` value

**Debugging Redirect Loops:**

```php
// Add to wp-config.php temporarily for debugging
error_log('HTTPS: ' . ($_SERVER['HTTPS'] ?? 'not set'));
error_log('REQUEST_URI: ' . $_SERVER['REQUEST_URI']);
error_log('HTTP_HOST: ' . $_SERVER['HTTP_HOST']);
error_log('SERVER_PORT: ' . $_SERVER['SERVER_PORT']);
error_log('X-Forwarded-Proto: ' . ($_SERVER['HTTP_X_FORWARDED_PROTO'] ?? 'not set'));
```

**Common Redirect Loop Causes:**

| Symptom | Cause | Solution |
|---------|-------|----------|
| Loop on all pages | WP_HOME/WP_SITEURL mismatch with actual URL | Verify constants match access URL |
| Loop on HTTPS only | `$_SERVER['HTTPS']` not set | Add proxy header detection |
| Loop on wp-admin | FORCE_SSL_ADMIN with HTTP access | Enable HTTPS or remove constant |
| Loop after migration | Old URLs in database | Run WP-CLI search-replace |

**Emergency Recovery:**

If locked out due to redirect loop, access via FTP/SSH and add to `wp-config.php`:

```php
// Temporary debugging - remove after fixing
define('WP_HOME', 'http://your-actual-domain.com');
define('WP_SITEURL', 'http://your-actual-domain.com');

// Disable all redirects temporarily
remove_action('template_redirect', 'redirect_canonical');
```

#### Mixed Content Warnings

After HTTPS migration, update:

```bash
# Update all internal URLs
wp search-replace 'http://example.com' 'https://example.com' --all-tables

# Check theme/plugin hardcoded URLs
wp search-replace 'http:' 'https:' wp_posts wp_postmeta --dry-run
```

**Mixed Content Detection Script:**

```bash
#!/bin/bash
# Find mixed content in database
echo "Checking for HTTP references in wp_posts..."

echo "Checking for HTTP references in wp_postmeta..."
wp db query "SELECT post_id, meta_key FROM wp_postmeta WHERE meta_value LIKE '%http://%'" --skip-column-names

echo "Checking for HTTP references in wp_options..."
wp db query "SELECT option_name FROM wp_options WHERE option_value LIKE '%http://%'" --skip-column-names
```

#### White Screen of Death (WSOD)

URL configuration errors can cause blank pages:

```php
// Add to wp-config.php for debugging
define('WP_DEBUG', true);
define('WP_DEBUG_LOG', true);
define('WP_DEBUG_DISPLAY', false);

// Check debug.log in wp-content/
// tail -f wp-content/debug.log
```

**Common WSOD Causes Related to URLs:**

| Error in Log | Cause | Solution |
|--------------|-------|----------|
| `Cannot modify header information` | Output before redirect | Check for whitespace before `<?php` |
| `Maximum execution time exceeded` | Infinite redirect loop | Fix WP_HOME/WP_SITEURL mismatch |
| `Allowed memory size exhausted` | Plugin conflict during URL change | Increase memory limit |

#### Database Connection Issues After Migration

After migrating to new server with different database host:

```php
// Update database host
define('DB_HOST', 'new-db-host.example.com');

// For Docker internal networking
define('DB_HOST', 'db');  // Service name in docker-compose

// For socket connections
```


### Best Practices

#### Configuration Management

1. **Always define both constants** together to avoid inconsistencies
2. **Use environment variables** for multi-environment deployments
3. **Document URL changes** in version control comments
4. **Test after changes** with `wp option get home` and `wp option get siteurl`
5. **Use HTTPS** in production environments
6. **Backup database** before direct SQL updates

#### Multi-Environment Deployment

**Environment Detection Pattern:**

```php
/**
 * Environment-based configuration
 * Detects environment from hostname or environment variable
 */

$environment = getenv('WP_ENV') ?: 'production';

// Auto-detect from hostname
if (!getenv('WP_ENV')) {
    $host = $_SERVER['HTTP_HOST'] ?? '';
    if (strpos($host, 'localhost') !== false || strpos($host, '.local') !== false) {
        $environment = 'development';
    } elseif (strpos($host, 'staging') !== false || strpos($host, 'test') !== false) {
        $environment = 'staging';
    }
}

switch ($environment) {
    case 'development':
        define('WP_HOME', 'http://localhost:8080');
        define('WP_SITEURL', 'http://localhost:8080');
        define('WP_DEBUG', true);
        define('WP_DEBUG_LOG', true);
        define('WP_DEBUG_DISPLAY', true);
        define('SCRIPT_DEBUG', true);
        break;

    case 'staging':
        define('WP_HOME', 'https://staging.example.com');
        define('WP_SITEURL', 'https://staging.example.com');
        define('WP_DEBUG', true);
        define('WP_DEBUG_LOG', true);
        define('WP_DEBUG_DISPLAY', false);
        break;

    case 'production':
    default:
        define('WP_HOME', 'https://example.com');
        define('WP_SITEURL', 'https://example.com');
        define('WP_DEBUG', false);
        define('DISALLOW_FILE_EDIT', true);
        break;
}
```

#### Monitoring and Validation

**Health Check Script:**

```bash
#!/bin/bash
# WordPress URL Health Check

SITE_URL=$(wp option get home --skip-plugins --skip-themes)
EXPECTED_URL="https://example.com"

echo "=== WordPress URL Health Check ==="
echo "Home URL:     $SITE_URL"
echo "Expected:     $EXPECTED_URL"

# Check HTTP response
HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" "$SITE_URL")
echo "HTTP Status:  $HTTP_CODE"

# Check for redirect loops
REDIRECT_COUNT=$(curl -s -o /dev/null -w "%{redirect_url}" -L --max-redirs 10 "$SITE_URL" 2>&1 | wc -l)
echo "Redirects:    $REDIRECT_COUNT"

# Check SSL certificate
if [[ "$SITE_URL" == https://* ]]; then
    SSL_EXPIRY=$(echo | openssl s_client -servername "${SITE_URL#https://}" -connect "${SITE_URL#https://}:443" 2>/dev/null | openssl x509 -noout -enddate 2>/dev/null)
    echo "SSL Expires:  ${SSL_EXPIRY#notAfter=}"
fi

# Validate URLs match
if [[ "$SITE_URL" == "$EXPECTED_URL" && "$HTTP_CODE" == "200" ]]; then
    echo "Status:       ✅ HEALTHY"
    exit 0
else
    echo "Status:       ❌ ISSUES DETECTED"
    exit 1
fi
```

**Emergency Recovery Steps:**

1. Access server via SSH/FTP
2. Edit `wp-config.php` with correct `WP_HOME` and `WP_SITEURL`
3. Clear `.htaccess` or rename to `.htaccess.bak`
4. Access wp-admin and verify Settings → General
5. Go to Settings → Permalinks and click "Save" to regenerate `.htaccess`
6. Test site thoroughly
