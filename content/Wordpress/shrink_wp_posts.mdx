## Overview

When managing WordPress installations, database bloat from accumulated posts, revisions, and metadata can significantly impact performance. This guide provides methods to safely reduce database size while preserving critical content and maintaining data integrity.

### Why Database Optimization Matters

- **Query Performance**: Smaller datasets result in faster query execution times
- **Backup Size**: Reduces backup file size and backup duration
- **Memory Usage**: InnoDB buffer pool operates more efficiently with less data
- **Server Resources**: Decreases disk I/O and CPU utilization

## Prerequisites

- Direct database access (MySQL/MariaDB)
- Database admin credentials
- Understanding of SQL DELETE operations
- **CRITICAL**: Full database backup before execution
- Read-only test environment recommended for validation

## Important Considerations

### Data Integrity

WordPress posts are interconnected with multiple related tables through foreign key relationships and referential integrity constraints:

| Related Table | Content | Impact | Relationship Type |
|---|---|---|---|
| `wp_postmeta` | Post metadata, custom fields | Orphaned records if parent deleted | One-to-Many (post_id) |
| `wp_comments` | Comments on posts | May reference deleted posts | One-to-Many (comment_post_ID) |
| `wp_term_relationships` | Category/tag associations | Foreign key constraints | Many-to-Many (object_id) |
| `wp_links` | Related post links | May break internal links | One-to-Many (link_target) |
| `wp_postmeta` | Custom fields, SEO metadata | Orphaned metadata records | One-to-Many (post_id) |
| `wp_term_relationships` | Post taxonomies | Breaks category/tag counts | Many-to-Many (object_id) |

**Warning**: Deleting posts orphans their associated comments and metadata unless handled with cascading deletes.

### InnoDB Storage Engine Characteristics

WordPress uses InnoDB by default, which has specific implications for bulk deletions:

**Transaction Isolation:**
- Default level: `REPEATABLE READ`
- DELETE operations acquire exclusive locks on affected rows
- Gap locks prevent phantom reads in range scans
- Lock escalation can occur with large batch operations

**Row-Level Locking Behavior:**
```sql
-- Check current transaction isolation level
SELECT @@transaction_ISOLATION;

-- View active InnoDB locks during deletion
SELECT * FROM performance_schema.data_locks
WHERE OBJECT_SCHEMA = 'wordpress_db'
AND OBJECT_NAME = 'wp_posts';

-- Monitor lock wait statistics
SELECT * FROM performance_schema.data_lock_waits;
```

**Buffer Pool Impact:**
- Deleted rows remain in buffer pool until evicted
- `OPTIMIZE TABLE` forces buffer pool refresh
- Monitor buffer pool efficiency:

```sql
-- Check InnoDB buffer pool statistics
SHOW ENGINE INNODB STATUS\G

-- Buffer pool hit ratio (target: >99%)
SELECT
    VARIABLE_VALUE AS buffer_pool_reads,
    (SELECT VARIABLE_VALUE FROM information_schema.GLOBAL_STATUS
     WHERE VARIABLE_NAME = 'Innodb_buffer_pool_read_requests') AS read_requests,
    ROUND(
        (1 - VARIABLE_VALUE /
         (SELECT VARIABLE_VALUE FROM information_schema.GLOBAL_STATUS
          WHERE VARIABLE_NAME = 'Innodb_buffer_pool_read_requests')) * 100,
        2
    ) AS hit_ratio_percent
FROM information_schema.GLOBAL_STATUS
WHERE VARIABLE_NAME = 'Innodb_buffer_pool_reads';
```

**Transaction Isolation Levels Explained:**

MySQL InnoDB supports four transaction isolation levels as defined by SQL standards. Understanding these is crucial for bulk deletion operations:

| Isolation Level | Dirty Reads | Non-Repeatable Reads | Phantom Reads | Performance | Use Case |
|---|---|---|---|---|---|
| `READ UNCOMMITTED` | Possible | Possible | Possible | Highest | Analytics (low accuracy requirements) |
| `READ COMMITTED` | Prevented | Possible | Possible | High | Most web applications |
| `REPEATABLE READ` (default) | Prevented | Prevented | Possible* | Medium | WordPress default (InnoDB gap locks) |
| `SERIALIZABLE` | Prevented | Prevented | Prevented | Lowest | Financial transactions |

*InnoDB uses gap locks in REPEATABLE READ to prevent phantom reads.

**Lock Types in DELETE Operations:**

```sql
-- Next-key locks: Combination of record lock + gap lock
-- Applied during DELETE operations in REPEATABLE READ isolation

-- Example: Deleting posts with ID > 100 and ID < 200
DELETE FROM wp_posts WHERE ID > 100 AND ID < 200;

-- InnoDB acquires:
-- 1. Record locks on existing rows (101, 102, ..., 199)
-- 2. Gap locks on ranges between rows
-- 3. Next-key locks = record lock + gap lock before record

-- View current lock waits
SELECT
    waiting_trx_id,
    waiting_pid,
    waiting_query,
    blocking_trx_id,
    blocking_pid,
    blocking_query,
    wait_started,
    wait_age,
    locked_table,
    locked_index,
    locked_type
FROM sys.innodb_lock_waits;

-- Check lock timeout setting
SHOW VARIABLES LIKE 'innodb_lock_wait_timeout';

-- Adjust lock timeout for bulk operations (default: 50 seconds)
SET SESSION innodb_lock_wait_timeout = 300;
```

**Deadlock Detection and Prevention:**

```sql
-- Enable deadlock detection (enabled by default)
SHOW VARIABLES LIKE 'innodb_deadlock_detect';

-- View recent deadlocks
SHOW ENGINE INNODB STATUS\G

-- Deadlock prevention strategies for bulk deletes:

-- Strategy 1: Order operations consistently
-- Always delete from parent to child tables in same order
DELETE FROM wp_postmeta WHERE post_id IN (SELECT ID FROM temp_delete_list);
DELETE FROM wp_term_relationships WHERE object_id IN (SELECT ID FROM temp_delete_list);
DELETE FROM wp_posts WHERE ID IN (SELECT ID FROM temp_delete_list);

-- Strategy 2: Use smaller transactions
-- Break large deletes into smaller batches
DELIMITER $$
CREATE PROCEDURE safe_batch_delete(IN batch_size INT, IN max_id INT)
BEGIN
    DECLARE current_id INT DEFAULT 0;
    DECLARE rows_deleted INT;

    delete_loop: LOOP
        START TRANSACTION;

        DELETE FROM wp_posts
        WHERE ID > current_id
        AND ID <= current_id + batch_size
        AND ID NOT IN (SELECT id FROM posts_to_keep)
        LIMIT 1000;

        SET rows_deleted = ROW_COUNT();
        SET current_id = current_id + batch_size;

        COMMIT;

        -- Exit if no more rows or reached max_id
        IF rows_deleted = 0 OR current_id >= max_id THEN
            LEAVE delete_loop;
        END IF;

        -- Brief pause to allow other transactions
        DO SLEEP(0.1);
    END LOOP;
END$$
DELIMITER ;

-- Execute the procedure
CALL safe_batch_delete(1000, 100000);

-- Monitor deadlock rate
SELECT
    VARIABLE_VALUE AS deadlock_count
FROM information_schema.GLOBAL_STATUS
WHERE VARIABLE_NAME = 'Innodb_deadlocks';
```

### MySQL Configuration for Bulk Operations

Before performing large-scale deletions, optimize MySQL configuration for bulk operations:

```sql
-- Temporarily increase delete buffer for large operations
SET GLOBAL innodb_change_buffering = 'all';

-- Adjust transaction commit behavior (use with caution)
-- 0 = Write and flush once per second (faster, less safe)
-- 1 = Write and flush at each commit (default, safest)

    SUBSYSTEM,

    COUNT
```

## Safe Deletion: Retain First 100 and Last 100 Posts

Delete posts in WordPress DB except 100 first and last 100

```sql
DELETE FROM wp_posts
WHERE ID NOT IN (
    SELECT ID FROM (
        -- Retain the first 100 rows
        SELECT ID FROM wp_posts ORDER BY ID ASC LIMIT 100
    ) AS first_100
    UNION
    SELECT ID FROM (
        -- Retain the last 100 rows
        SELECT ID FROM wp_posts ORDER BY ID DESC LIMIT 100
    ) AS last_100
);
```

### Understanding the Query Mechanics

This query uses a **derived table pattern** to circumvent MySQL's limitation where you cannot directly reference the same table in a DELETE and subquery. The nested `SELECT` statements create temporary result sets that MySQL treats as separate entities.

**Key Technical Points:**

- **Subquery Execution Order**: MySQL first evaluates the innermost `SELECT` statements to build the `first_100` and `last_100` derived tables
- **UNION Operator**: Combines both result sets while automatically removing duplicates (critical if a database has fewer than 200 posts)
- **NOT IN Performance**: For small result sets (&lt;1000 rows), `NOT IN` performs adequately. For larger operations, consider `LEFT JOIN ... WHERE ... IS NULL` pattern
- **Transaction Isolation**: This operation acquires row-level locks on InnoDB tables during execution

For long-running operations, monitor progress and resource usage:

```sql

CREATE TEMPORARY TABLE deletion_progress (
```

### Batch Deletion for Large Datasets

For databases with millions of posts, deleting in batches prevents long-running locks and replication lag:

```sql
-- Delete in batches of 1000 until complete
SET @batch_size = 1000;
SET @rows_affected = 1;

WHILE @rows_affected > 0 DO
    DELETE FROM wp_posts
    WHERE ID NOT IN (
        SELECT ID FROM (
            SELECT ID FROM wp_posts ORDER BY ID ASC LIMIT 100
        ) AS first_100
        UNION
        SELECT ID FROM (
            SELECT ID FROM wp_posts ORDER BY ID DESC LIMIT 100
        ) AS last_100
    )
    LIMIT @batch_size;

    SET @rows_affected = ROW_COUNT();

    -- Pause to allow replication catchup and reduce I/O pressure
    SELECT SLEEP(0.5);
END WHILE;
```

### Cascading Cleanup of Related Data

After post deletion, clean up orphaned records in related tables:

```sql
-- Remove orphaned post metadata
DELETE pm FROM wp_postmeta pm
LEFT JOIN wp_posts p ON pm.post_id = p.ID
WHERE p.ID IS NULL;

-- Remove orphaned term relationships
DELETE tr FROM wp_term_relationships tr
LEFT JOIN wp_posts p ON tr.object_id = p.ID
WHERE p.ID IS NULL;

-- Update comment counts for posts
UPDATE wp_posts p
SET comment_count = (
    SELECT COUNT(*)
    FROM wp_comments c
    WHERE c.comment_post_ID = p.ID
    AND c.comment_approved = '1'
);

-- Optional: Delete orphaned comments
DELETE c FROM wp_comments c
LEFT JOIN wp_posts p ON c.comment_post_ID = p.ID
WHERE p.ID IS NULL;
```

### Database Health and Statistics Analysis

After cleanup operations, perform comprehensive health checks:

```sql
-- Comprehensive table statistics
SELECT
    table_name,
    engine,
    table_rows,
    ROUND(data_length / 1024 / 1024, 2) AS data_mb,
    ROUND(index_length / 1024 / 1024, 2) AS index_mb,
    ROUND((data_length + index_length) / 1024 / 1024, 2) AS total_mb,
    ROUND(data_free / 1024 / 1024, 2) AS free_mb,
    ROUND((data_free / (data_length + index_length + data_free)) * 100, 2) AS fragmentation_percent
FROM information_schema.tables
WHERE table_schema = DATABASE()
AND table_name IN ('wp_posts', 'wp_postmeta', 'wp_comments', 'wp_term_relationships')
ORDER BY (data_length + index_length) DESC;

-- Index cardinality analysis
SELECT
    table_name,
    index_name,
    non_unique,
    seq_in_index,
    column_name,
    cardinality,
    CASE
        WHEN cardinality IS NULL THEN 'N/A'
        WHEN cardinality = 0 THEN 'Poor'
        WHEN cardinality < 100 THEN 'Low'
        WHEN cardinality < 1000 THEN 'Medium'
        ELSE 'High'
    END AS selectivity
FROM information_schema.statistics
WHERE table_schema = DATABASE()
AND table_name = 'wp_posts'
ORDER BY table_name, index_name, seq_in_index;

-- Check for duplicate or redundant indexes
SELECT
    a.table_name,
    a.index_name AS index_1,
    GROUP_CONCAT(a.column_name ORDER BY a.seq_in_index) AS columns_1,
    b.index_name AS index_2,
    GROUP_CONCAT(b.column_name ORDER BY b.seq_in_index) AS columns_2
FROM information_schema.statistics a
JOIN information_schema.statistics b ON a.table_schema = b.table_schema
    AND a.table_name = b.table_name
    AND a.index_name != b.index_name
    AND a.column_name = b.column_name
    AND a.seq_in_index = b.seq_in_index
WHERE a.table_schema = DATABASE()
AND a.table_name = 'wp_posts'
GROUP BY a.table_name, a.index_name, b.index_name
HAVING columns_1 = columns_2;

-- Identify orphaned metadata across all post-related tables
SELECT
    'wp_postmeta' AS table_name,

WHERE p.ID IS NULL

UNION ALL

SELECT
    'wp_comments' AS table_name,
    COUNT(*) AS orphaned_records,
    ROUND(SUM(LENGTH(comment_content)) / 1024 / 1024, 2) AS wasted_mb
FROM wp_comments c
LEFT JOIN wp_posts p ON c.comment_post_ID = p.ID
WHERE p.ID IS NULL

UNION ALL

SELECT
    'wp_term_relationships' AS table_name,
    COUNT(*) AS orphaned_records,
    0 AS wasted_mb
FROM wp_term_relationships tr
LEFT JOIN wp_posts p ON tr.object_id = p.ID
WHERE p.ID IS NULL;

-- Analyze post type distribution and size
SELECT
    post_type,
    post_status,
    COUNT(*) AS post_count,
    ROUND(AVG(LENGTH(post_content)), 2) AS avg_content_length,
    ROUND(SUM(LENGTH(post_content)) / 1024 / 1024, 2) AS total_content_mb,
    MIN(post_date) AS oldest_post,
    MAX(post_date) AS newest_post
FROM wp_posts
GROUP BY post_type, post_status
ORDER BY post_count DESC;
```

```sql
    s.COLUMN_NAME,
    s.SEQ_IN_INDEX AS column_position,
    s.CARDINALITY,
    ROUND((s.CARDINALITY / t.TABLE_ROWS) * 100, 2) AS selectivity_percent,
    CASE
        WHEN u.INDEX_NAME IS NULL THEN 'UNUSED - Consider dropping'
        ELSE 'USED'
    END AS usage_status
FROM information_schema.STATISTICS s
INNER JOIN information_schema.TABLES t
    ON s.TABLE_SCHEMA = t.TABLE_SCHEMA
    AND s.TABLE_NAME = t.TABLE_NAME
LEFT JOIN (
    SELECT DISTINCT OBJECT_SCHEMA, OBJECT_NAME, INDEX_NAME
    FROM performance_schema.table_io_waits_summary_by_index_usage
    WHERE COUNT_STAR > 0
) u ON s.TABLE_SCHEMA = u.OBJECT_SCHEMA
    AND s.TABLE_NAME = u.OBJECT_NAME
    AND s.INDEX_NAME = u.INDEX_NAME
WHERE s.TABLE_SCHEMA = DATABASE()
AND s.TABLE_NAME = 'wp_posts'
AND s.INDEX_NAME != 'PRIMARY'
ORDER BY t.TABLE_NAME, s.INDEX_NAME, s.SEQ_IN_INDEX;

-- Check for duplicate indexes
SELECT
    a.TABLE_NAME,
    a.INDEX_NAME AS index1,
    GROUP_CONCAT(a.COLUMN_NAME ORDER BY a.SEQ_IN_INDEX) AS columns1,
    b.INDEX_NAME AS index2,
    GROUP_CONCAT(b.COLUMN_NAME ORDER BY b.SEQ_IN_INDEX) AS columns2,
    'Potential duplicate - review for removal' AS recommendation
FROM information_schema.STATISTICS a
JOIN information_schema.STATISTICS b
    ON a.TABLE_SCHEMA = b.TABLE_SCHEMA
    AND a.TABLE_NAME = b.TABLE_NAME
    AND a.INDEX_NAME < b.INDEX_NAME
WHERE a.TABLE_SCHEMA = DATABASE()
AND a.TABLE_NAME = 'wp_posts'
GROUP BY a.TABLE_NAME, a.INDEX_NAME, b.INDEX_NAME
HAVING columns1 = columns2;

-- Rebuild indexes to reclaim space and improve performance
ALTER TABLE wp_posts ENGINE=InnoDB;

-- Or rebuild specific index
ALTER TABLE wp_posts DROP INDEX type_status_date;
ALTER TABLE wp_posts ADD INDEX type_status_date (post_type, post_status, post_date, ID);

-- Force index hints for queries
-- Use when optimizer makes poor choices after deletion
SELECT * FROM wp_posts FORCE INDEX (type_status_date)
WHERE post_type = 'post'
AND post_status = 'publish'
ORDER BY post_date DESC
LIMIT 10;
```

**Index Best Practices for WordPress:**

**Composite Index Ordering:** Most selective columns first

```sql
-- Good: Filters reduce dataset quickly
INDEX (post_type, post_status, post_date)

-- Bad: Less selective column first
INDEX (post_date, post_type, post_status)
```

**Covering Indexes:** Include all columns in SELECT

```sql
-- Query: SELECT post_title, post_date FROM wp_posts WHERE post_type = 'post'
-- Covering index includes all needed columns
CREATE INDEX idx_covering ON wp_posts(post_type, post_date, post_title);
```

**Prefix Indexes:** For long VARCHAR/TEXT columns

```sql
-- Only index first 50 characters of post_name
CREATE INDEX idx_post_name_prefix ON wp_posts(post_name(50));
```

## Troubleshooting

### Common Issues and Solutions

**Issue**: Query times out on large databases

**Solution**: Use batch deletion with `LIMIT` and iterate:

```sql
-- Incremental deletion
SET @deleted = 1;
WHILE @deleted > 0 DO
    DELETE FROM wp_posts
    WHERE ID NOT IN (
        SELECT ID FROM (
            SELECT ID FROM wp_posts ORDER BY ID ASC LIMIT 100
        ) AS first_100
        UNION
        SELECT ID FROM (
            SELECT ID FROM wp_posts ORDER BY ID DESC LIMIT 100
        ) AS last_100
    )
    LIMIT 5000;
    SET @deleted = ROW_COUNT();
    SELECT @deleted AS 'Rows deleted in this batch';
END WHILE;
```

**Issue**: Replication lag in master-slave setup

**Solution**: Add delays between batch operations:

```sql
DELETE FROM wp_posts WHERE [...] LIMIT 1000;
SELECT SLEEP(2); -- 2 second pause
```

**Issue**: Foreign key constraints prevent deletion

**Solution**: Temporarily disable foreign key checks (use with extreme caution):

```sql
SET FOREIGN_KEY_CHECKS = 0;
-- Perform deletion
DELETE FROM wp_posts WHERE [...];
SET FOREIGN_KEY_CHECKS = 1;
```

```php
<?php
/**
 * Safe bulk post deletion using WordPress APIs
 * Respects post relationships and triggers proper hooks
 */
function safely_delete_old_posts($keep_first = 100, $keep_last = 100) {
    global $wpdb;

    // Get total post count
    $total = $wpdb->get_var("SELECT COUNT(*) FROM {$wpdb->posts} WHERE post_type = 'post'");

    if ($total <= ($keep_first + $keep_last)) {
        return new WP_Error('insufficient_posts', 'Not enough posts to delete');
    }

    // Get IDs to keep
    $keep_ids = array_merge(
        $wpdb->get_col($wpdb->prepare(
            "SELECT ID FROM {$wpdb->posts}
             WHERE post_type = 'post'
             ORDER BY ID ASC LIMIT %d",
            $keep_first
        )),
        $wpdb->get_col($wpdb->prepare(
            "SELECT ID FROM {$wpdb->posts}
             WHERE post_type = 'post'
             ORDER BY ID DESC LIMIT %d",
            $keep_last
        ))
    );

    // Get posts to delete
    $delete_ids = $wpdb->get_col($wpdb->prepare(
        "SELECT ID FROM {$wpdb->posts}
         WHERE post_type = 'post'
         AND ID NOT IN (" . implode(',', array_map('intval', $keep_ids)) . ")"
    ));

    $deleted_count = 0;

    // Delete in batches of 50
    foreach (array_chunk($delete_ids, 50) as $batch) {
        foreach ($batch as $post_id) {
            // wp_delete_post handles metadata, comments, and relationships
            if (wp_delete_post($post_id, true)) {
                $deleted_count++;
            }
        }

        // Prevent memory exhaustion
        wp_cache_flush();

        // Give server breathing room
        sleep(1);
    }

    return $deleted_count;
}
```

### Automated Health Checks

Create stored procedures for recurring health checks:

```sql
DELIMITER $$

CREATE PROCEDURE check_database_health()
BEGIN
    -- Check for orphaned records
    SELECT
        'Orphaned postmeta' AS check_type,
        COUNT(*) AS issues_found,
        CASE
            WHEN COUNT(*) = 0 THEN 'PASS'
            WHEN COUNT(*) < 100 THEN 'WARNING'
            ELSE 'CRITICAL'
        END AS status
    FROM wp_postmeta pm
    LEFT JOIN wp_posts p ON pm.post_id = p.ID
    WHERE p.ID IS NULL

    UNION ALL

    -- Check table fragmentation
    SELECT
        'Table fragmentation' AS check_type,
        ROUND((data_free / (data_length + index_length + data_free)) * 100, 2) AS issues_found,
        CASE
            WHEN (data_free / (data_length + index_length + data_free)) * 100 < 10 THEN 'PASS'
            WHEN (data_free / (data_length + index_length + data_free)) * 100 < 25 THEN 'WARNING'
            ELSE 'CRITICAL'
        END AS status
    FROM information_schema.tables
    WHERE table_schema = DATABASE()
    AND table_name = 'wp_posts'

    UNION ALL

    -- Check InnoDB buffer pool usage
    SELECT
        'Buffer pool usage' AS check_type,
        ROUND(
            (SELECT VARIABLE_VALUE FROM information_schema.GLOBAL_STATUS
             WHERE VARIABLE_NAME = 'Innodb_buffer_pool_pages_data') /
            (SELECT VARIABLE_VALUE FROM information_schema.GLOBAL_STATUS
             WHERE VARIABLE_NAME = 'Innodb_buffer_pool_pages_total') * 100,
            2
        ) AS issues_found,
        CASE
            WHEN (SELECT VARIABLE_VALUE FROM information_schema.GLOBAL_STATUS
                  WHERE VARIABLE_NAME = 'Innodb_buffer_pool_pages_data') /
                 (SELECT VARIABLE_VALUE FROM information_schema.GLOBAL_STATUS
                  WHERE VARIABLE_NAME = 'Innodb_buffer_pool_pages_total') * 100 < 80 THEN 'PASS'
            WHEN (SELECT VARIABLE_VALUE FROM information_schema.GLOBAL_STATUS
                  WHERE VARIABLE_NAME = 'Innodb_buffer_pool_pages_data') /
                 (SELECT VARIABLE_VALUE FROM information_schema.GLOBAL_STATUS
                  WHERE VARIABLE_NAME = 'Innodb_buffer_pool_pages_total') * 100 < 95 THEN 'WARNING'
            ELSE 'CRITICAL'
        END AS status;
END$$

DELIMITER ;
```

```sql
-- Monitor memory usage
SELECT
    SUBSTRING_INDEX(event_name, '/', 2) AS memory_category,
    ROUND(SUM(CURRENT_NUMBER_OF_BYTES_USED) / 1024 / 1024, 2) AS memory_used_mb
FROM performance_schema.memory_summary_global_by_event_name
WHERE event_name LIKE 'memory/%'
GROUP BY memory_category
ORDER BY SUM(CURRENT_NUMBER_OF_BYTES_USED) DESC
LIMIT 10;
```

### Issue: Disk Space Not Reclaimed

**Symptoms:** Table size doesn't decrease after deletion

**Solutions:**
```sql
-- Check fragmentation
SELECT
    table_name,
    ROUND(data_free / 1024 / 1024, 2) AS free_space_mb,
    ROUND((data_free / (data_length + index_length + data_free)) * 100, 2) AS fragmentation_percent
FROM information_schema.tables
WHERE table_schema = DATABASE()
AND table_name = 'wp_posts';

-- Reclaim space
OPTIMIZE TABLE wp_posts;

-- Verify file-per-table mode
SHOW VARIABLES LIKE 'innodb_file_per_table';
```
